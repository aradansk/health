# Phase 1.0: Fasten Ingest API Spike - Research

**Researched:** 2026-05-19
**Domain:** Fasten OnPrem (Go/gin FHIR aggregator) ingest API surface + container PII-log audit
**Confidence:** HIGH (route surface, auth model, container config verified directly from `main` source via `gh api`; PII-log behaviour MEDIUM — derived from logrus usage patterns, must be confirmed empirically in the spike)

## Summary

The single highest-risk MEDIUM-confidence assumption for the whole M1 build — "does Fasten expose a programmatic FHIR Bundle ingest path?" — **resolves to YES with HIGH confidence**. Source inspection of `fastenhealth/fasten-onprem@main` shows a fully usable HTTP API: `POST /api/secure/source/manual` accepts a `multipart/form-data` upload (field name `file`) containing a FHIR Bundle JSON and ingests every resource in it. This is the *same* endpoint the Fasten UI's "manual source / upload bundle" flow calls — it is not UI-exclusive, it is a normal authenticated REST route. Authentication is `Authorization: Bearer <JWT>`. Critically, Fasten ships a **first-class long-lived API token mechanism** (`POST /api/secure/access/token`, designed for the mobile app) that returns a JWT with `TokenType=access` and a configurable multi-year expiry — purpose-built for exactly this ETL use case, far more robust than scraping a session cookie.

This means the project does **not** need any of the fragile fallbacks (volume-mount JSON drop / direct SQLite write / UI scrape). Those are documented below as ranked contingencies only — the spike's expected outcome is to confirm the primary path empirically and lock it in `docs/fasten-admin.md`.

The paired PII-log deliverable: Fasten uses `logrus` with a configurable level (`log.level`, default `INFO`; settable to `WARN`). The ingest/sync path logs via `logrus.Entry` and on error returns `err.Error()` strings that can echo bundle-derived content. There are no open GitHub issues specifically flagging PII-in-logs, so the audit must be **empirical**: ingest a synthetic known-token Bundle, grep `docker logs`, decide on `LOG_LEVEL=WARN` and/or a restricted log mount.

**Primary recommendation:** Spike confirms `POST /api/secure/source/manual` with an `access` token; pin `ghcr.io/fastenhealth/fasten-onprem:main@sha256:3f0192ac77dda7fd0e25175f7383a426f6d5ee11be7981b8753c10f6b7447d91` (multi-arch OCI index digest — correct for Win-dev → Linux-prod parity); document endpoint + token-mint + response semantics + LOG_LEVEL decision in `docs/fasten-admin.md`.

### Pattern 1: Mint a long-lived `access` token (the ETL auth pattern)
**What:** Two-step auth — sign in with admin creds to get a 1h session JWT, then exchange it for a long-lived `access` JWT that the ETL stores in Vaultwarden.
**When to use:** Every ETL writer in Phase 1.4+. Never embed the password in the ETL; mint the token once, store via `bw`.
**Example:**
```python
# Source: github.com/fastenhealth/fasten-onprem/blob/main/backend/pkg/web/handler/auth.go
#         + handler/access_token.go + middleware/require_auth.go  [VERIFIED: gh api main]
import httpx

BASE = "http://localhost:8080"
s = httpx.post(f"{BASE}/api/auth/signin",
               json={"username": ADMIN_USER, "password": ADMIN_PASS}).json()
session_jwt = s["data"]                       # success envelope: {"success":true,"data":<jwt>}

t = httpx.post(f"{BASE}/api/secure/access/token",
               headers={"Authorization": f"Bearer {session_jwt}"},
               json={"name": "etl-andrej", "expiration": 0}).json()  # 0 => exp 2099-12-31
access_jwt = t["data"]                          # long-lived; store via `bw get`
```

### Pattern 2: Ingest a FHIR Bundle (the primary ingest path)
**What:** `multipart/form-data` POST, single form field `file` holding the Bundle JSON.
**Example:**
```python
# Source: github.com/fastenhealth/fasten-onprem/blob/main/backend/pkg/web/handler/source.go
#         CreateManualSource: file,_ := c.FormFile("file"); ... SyncAllBundle(...)  [VERIFIED: gh api main]
r = httpx.post(f"{BASE}/api/secure/source/manual",
               headers={"Authorization": f"Bearer {access_jwt}"},
               files={"file": ("bundle.json", bundle_json_bytes, "application/json")})
# 200 => {"success":true,"data":<sync summary>,"source":<manualSourceCredential>}
# 500 => {"success":false,"error":"<err.Error()>"}   (only 200 or 500 — no 4xx for body errors)
```

### Anti-Patterns to Avoid
- **Re-authenticating per ETL run with the password:** session JWT is 1h; mint an `access` token once, store in Vaultwarden, rotate per the 8-week re-pin runbook.
- **Treating manual-source POST as idempotent:** it is *not*. Each POST creates a new manual source + ingests all resources. Idempotency must be enforced by the ETL via `meta.tag.code = sha256(canonical_payload)` (DATA-09) — confirmed as a Phase 1.4 concern, but the spike should note whether a re-POST of the identical Bundle produces duplicate resources (expected: yes → dedup is ETL's job, not Fasten's).
- **Pinning the per-arch digest:** pin the **OCI index** digest (multi-arch) so Win-dev (amd64) and Linux-prod (amd64/arm64) resolve correctly from one pin (INFRA-10 cross-host parity).
- **Leaving `jwt.issuer.key` default:** Fasten's config default is the literal string `thisismysupersecuressessionsecretlength`. If unset, **anyone can forge a valid admin JWT**. The spike MUST set `FASTEN_JWT_ISSUER_KEY` (or config) to a Vaultwarden-sourced secret and the finding MUST be flagged for Phase 1.1/1.4.

## Common Pitfalls

### Pitfall 1: Insecure default JWT signing key
**What goes wrong:** Fasten's `jwt.issuer.key` defaults to a published constant string. With it unchanged, any party can forge a valid `admin` JWT and POST/read all FHIR data.
**Why it happens:** Config default exists for dev convenience; easy to never override in a quick spike.
**How to avoid:** Spike sets a unique key via env/config. Document in `docs/fasten-admin.md` as a **mandatory Phase 1.1 secret** (Vaultwarden-sourced, `FASTEN_JWT_ISSUER_KEY`).
**Warning signs:** Fasten boots without complaint using the default — there is no startup warning; absence of error is NOT safety.
**Confidence:** HIGH — `c.SetDefault("jwt.issuer.key", "thisismysupersecuressessionsecretlength")` `[VERIFIED: gh api main backend/pkg/config/config.go]`

### Pitfall 2: PII in Fasten stdout logs (the paired deliverable)
**What goes wrong:** Fasten logs via `logrus`; the ingest/sync path holds a `logrus.Entry`, and error responses return raw `err.Error()` strings that can contain bundle-derived content. `docker logs` + host json-file driver persist this outside the project's pino redaction allowlist (SEC-02 covers only the custom layer, not Fasten).
**Why it happens:** Fasten is a third party; its logging is not designed around the project's PII Tier 1 baseline. At `INFO` (default) sync detail is verbose.
**How to avoid (audit method, success criterion 2):**
1. Ingest a Bundle whose only "PII" is a unique sentinel (`ZZZ-PII-CANARY-0001` in a Patient name + an Observation note).
2. `docker logs health-fasten-spike 2>&1 | Select-String "ZZZ-PII-CANARY"` (PowerShell) or `| grep` (bash). Repeat at `LOG_LEVEL=INFO` and `LOG_LEVEL=WARN`.
3. Record findings + mitigation in `docs/fasten-admin.md`: set `LOG_LEVEL=WARN`; Docker `local` driver with rotation (OPS-07 `max-size:10m max-file:3`); restricted host log mount; redaction sidecar (vector/fluent-bit) only if WARN still leaks.
**Warning signs:** Canary string appears in `docker logs` at any level.
**Confidence:** MEDIUM — logrus usage + `err.Error()` echo verified in source; *whether the canary actually surfaces* must be confirmed empirically (that is the spike).

### Pitfall 3: Floating `:main` tag drift
**What goes wrong:** `:main` is rebuilt on every upstream push; an un-pinned tag means the spike's verified contract silently diverges from what Phase 1.4 runs.
**How to avoid:** Pin the multi-arch index digest (below). Phase 1.11 owns the 8-week re-pin runbook (INFRA-09).
**Confidence:** HIGH.

### Pitfall 4: Manual-source ingest is not idempotent
**What goes wrong:** Re-POSTing the same Bundle creates a *new* manual source and re-ingests — duplicates in Fasten UI.
**How to avoid:** Out of scope to *fix* in 1.0; the spike must *document the behaviour* so Phase 1.4's `meta.tag.code = sha256(...)` dedup design (DATA-09) is grounded in fact, not assumption. Add one extra spike step: POST the identical Bundle twice, record resource count delta.
**Confidence:** MEDIUM (inferred from `CreateManualSource` creating a fresh `manualSourceCredential` each call; confirm empirically).

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|

## Project Constraints (from CLAUDE.md)

- **PII Tier 1, strict:** spike MUST use only synthetic sentinel values; never ingest or invent real health data. No PII into logs, AI payloads, or git.
- **GPL-3.0 architectural firewall (COMPL-06):** custom code talks to Fasten over HTTP only; **no patching Fasten source**, no shared-lib linkage. The PII-log mitigation must therefore be infra-layer (Docker driver / sidecar / LOG_LEVEL), never a Fasten code change. The spike *reads* Fasten source for understanding only — produces no fork.
- **Secrets via Vaultwarden:** the production Fasten admin password + `jwt.issuer.key` + `access` token come from `bw get default/<secret>` (Phase 1.1/1.4). The spike uses *throwaway* values, never the real production secret.
- **Image pinning (INFRA-06):** `:main@sha256:<digest>`, never `:latest` / `v1.1.3` / floating `:main`.
- **Data isolation:** spike artifacts (`sample_bundle.json`, `docker logs` excerpts) must contain no real medical data before any commit; `docs/fasten-admin.md` documents the *contract*, never sample patient data.
- **Vault writes:** product-knowledge (the ingest contract) is eligible for `SOLO/health/Architecture/`; never real health values. Update `Knowledge/AGENT-INDEX.md` after any vault write (existence + category only).
