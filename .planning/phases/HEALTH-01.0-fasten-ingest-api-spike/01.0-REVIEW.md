# Phase HEALTH-01.0: Code Review Report

**Reviewed:** 2026-05-19
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

`docs/fasten-admin.md` is the single durable deliverable of the Fasten Ingest API
spike — a source-of-truth contract that Phase 1.4+ ETL will implement against. It
was reviewed as a security/accuracy-critical specification, not as prose.

The document is largely strong: the auth/ingest contract is internally consistent
on endpoint paths and the multipart field name; epoch arithmetic was independently
verified (`exp 4102444799` = 2099-12-31T23:59:59Z, `iat 1779192818` = 2026-05-19,
both correct); the `jwt.issuer.key` insecure default is prominently flagged as a
mandatory Vaultwarden secret (§3); the INFRA-06 digest drift is recorded without
silent re-pinning (§2.1); and the escalation/fallback section (§5, §6) is present
and actionable. No real PII or project credential leaks — only the permitted
synthetic sentinel and the *upstream-published* default constant appear.

However, the document contains one **Critical** internal contradiction in its own
core security audit: §3 explicitly documents that the mandatory SQLCipher
`database.encryption.key` is echoed verbatim into the Fasten boot log
(`_key=<key>`), yet the §4 PII/secret-log audit only greps for the data sentinel
(`ZZZ-PII-CANARY`), never for the secret, and then concludes the logs are
"PII-clean". For a PII-Tier-1 / Vaultwarden-secret project, a mandatory secret
landing in `docker logs` is a secret-disclosure finding the contract under-states
and propagates to Phase 1.4 as a false "logs are clean" assumption. Plus
contract-level inconsistencies in the token-mint payload.

## Critical Issues

### CR-01: Mandatory SQLCipher encryption key is logged in cleartext to `docker logs`; the §4 "PII-clean" audit never checks for it

**File:** `docs/fasten-admin.md:198` (claim), `docs/fasten-admin.md:210-230` (audit that misses it), `docs/fasten-admin.md:232-240` (mitigations that don't cover it)

**Issue:**
§3 line 198 states verbatim: *"Boot log confirms SQLCipher: `_cipher=sqlcipher … _key=<key>`."* This means Fasten writes the **mandatory Vaultwarden-sourced `database.encryption.key`** — whose loss is documented one line earlier as making the DB *"unrecoverable"* — into its own boot log, which goes to `docker logs`.

§4.1 (the doc's self-described *"core security finding"*) audits log leakage by running only:
`docker logs health-fasten-spike 2>&1 | Select-String "ZZZ-PII-CANARY"` (lines 213, 343). It greps exclusively for the *data* sentinel and concludes (line 220-222) *"at neither INFO nor WARN did bundle-derived content … leak"* and (line 230) the logs are *"PII-clean for the ingest path."* The §4.1 mitigation list (lines 232-240) addresses payload redaction, rotation, and retention but **never addresses the secret already known to be in the boot log**. CLAUDE.md is explicit: *"Heslá / API tokeny … NIKDY do logov"* and *"Encryption-at-rest … DB backup encrypted."* A secret in `docker logs` survives in the Docker json/local log files and any future log shipping, defeating the Vaultwarden-sourcing the contract mandates in §3.

Because this file is the source-of-truth contract, Phase 1.4 will inherit the unqualified conclusion "logs are PII-clean" and may ship log mounts / shipping without redacting the encryption key (and likely the JWT issuer key, which the same config-load path may also echo — untested).

**Fix:** Add an explicit secret-in-log finding and acceptance gate. Concretely, in §4.1:

```markdown
### 4.1.b SECRET-in-log finding (boot log, NOT ingest-path)

`database.encryption.key` (and possibly `jwt.issuer.key`) IS emitted to the boot
log via the SQLCipher DSN: `_cipher=sqlcipher … _key=<key>` (§3). The canary audit
above only covers the *ingest data path*; it does NOT cover startup secrets.

Verdict: SECRET LEAK at startup. `docker logs` is NOT secret-clean.

Phase 1.4 MANDATORY mitigations (in addition to §4.1 list):
- Treat the Fasten container log as secret-bearing: restricted mount, no shipping
  of boot lines, ≤ retention, never into AI/debug payloads (CLAUDE.md PII T1).
- Re-run the audit grepping for the actual key value AND `_key=` / `issuer.key`
  at INFO and WARN; record occurrence counts the same way §4.1 does for the canary.
- Acceptance gate: assert the Vaultwarden key value does NOT appear in any log
  destination that leaves the host.
```

Also soften the §4.1 headline conclusion from "logs are PII-clean" to "the *ingest data path* is PII-clean; the *boot path* is NOT secret-clean — see §4.1.b".

## Warnings

### WR-01: Token-mint contract payload is inconsistent between §0 and §1.1 (`name` value)

**File:** `docs/fasten-admin.md:18` vs `docs/fasten-admin.md:59`

**Issue:** The TL;DR contract (line 18) specifies the access-token mint body as
`{"name":"etl","expiration":0}`, while the authoritative §1.1 Step B (line 59)
specifies `{"name":"etl-spike","expiration":0}`. A contract document with two
different request bodies for the same call forces Phase 1.4 to guess which is
canonical. (The decoded sample at line 66 has `"sub":"spikeadmin"`, which does not
disambiguate `name`.) Minor functionally (the `name` is a label), but this is a
*contract* — divergent examples erode its authority.

**Fix:** Pick one canonical value (recommend a stable production-intent name, e.g.
`"name":"etl"`, not the throwaway `"etl-spike"`) and use it identically in §0 and
§1.1, or add a one-line note that `name` is a free-form label with no semantic
effect on the issued token.

### WR-02: The doc prints the upstream default `jwt.issuer.key` verbatim without a "do not copy this into any real config" guard

**File:** `docs/fasten-admin.md:184-185`

**Issue:** §3 reproduces the literal default value
`thisismysupersecuressessionsecretlength`. This is the *publicly published upstream
constant*, so it is not a project-secret leak — and printing it makes the "anyone
can forge a JWT" warning concrete and credible, which is good. The risk is the
opposite: a contract that literally contains a working-format secret string can be
copy-pasted by a hurried Phase 1.4 implementer into a `config.yaml` template, and
it would *appear* to work (no startup warning — the doc itself notes Fasten emits
none, line 189). The doc mandates Vaultwarden-sourcing but does not put a hard
inline guard next to the string itself.

**Fix:** Annotate the literal so it cannot be copied as-is, e.g.:
`thisismysupersecuressessionsecretlength`  ⚠️ **NEVER place this (or any literal)
in a deployed config — `jwt.issuer.key` MUST be `{{ bw get … }}` templated (§3).
This value is shown only to prove the forgeable-default exists.**

### WR-03: Idempotency finding documents unbounded `manualSourceCredential` row growth with no spike-side bound check

**File:** `docs/fasten-admin.md:250-263`

**Issue:** §4.2 establishes that every `POST /api/secure/source/manual` creates a
**new `manualSourceCredential` row** (non-idempotent at the source layer), and
correctly pushes a `meta.tag.code = sha256(...)` dedup requirement to Phase 1.4
DATA-09. However, the contract gives Phase 1.4 no documented signal about the
*blast radius*: an ETL cron that re-syncs (e.g. Apple Health daily) will accrete a
source row per run indefinitely, and the doc's own §4.1 access-log line shows each
POST is logged. There is no documented expectation for whether stale source rows
must be reaped, nor whether the verify `GET /api/secure/resource/fhir` count is
affected by source-row proliferation over many runs (the spike only ran `--twice`).
A source-of-truth contract that surfaces a non-idempotent write should state the
unmitigated growth consequence, not only the dedup mitigation.

**Fix:** Add to the §4.2 Phase 1.4 implication: *"Unmitigated, source-row growth is
O(number of ETL runs) and is NOT bounded by Fasten. Phase 1.4 DATA-09 must (a)
content-hash-guard before POST AND (b) define a stale `manualSourceCredential`
reaping/limit strategy; the spike only validated n=2, not steady-state."*

## Info

### IN-01: §1.1 documents a ~73-year (effectively non-expiring) access token with no server-side revocation path discussed

**File:** `docs/fasten-admin.md:18, 59, 71`

**Issue:** `expiration:0 ⇒ exp = 2099-12-31` mints a token that for all practical
purposes never expires. §3 does mandate Vaultwarden storage + 8-week rotation
(good, mitigation present), but the contract never states whether Fasten supports
*server-side* revocation of an already-issued access JWT (if the stored token
leaks before the 8-week rotation, is there a `DELETE`/revoke route, or is the only
remedy rotating `jwt.issuer.key` — which invalidates *all* tokens?). For a token
this long-lived, Phase 1.4 needs that fact.

**Fix:** Add a one-line note in §3: whether per-token revocation exists; if not,
state explicitly that the only revocation is rotating `jwt.issuer.key` (mass
invalidation) and that this is acceptable given single-user MVP, revisit at SaaS.

### IN-02: §4.1 leak audit covers only INFO and WARN levels; DEBUG/TRACE untested

**File:** `docs/fasten-admin.md:216-219`

**Issue:** The PII-leak table tests only `INFO` (default) and `warn`. The doc
correctly flags (line 240) that PII-leak behaviour is build-specific and must be
re-run on digest bump, but does not note that a more verbose level (debug/trace),
if ever enabled for troubleshooting in Phase 1.4, is *unaudited* and may surface
payload. Low risk given the chosen mitigation is `log.level: warn`, but worth a
contract caveat.

**Fix:** Add a row or footnote: *"DEBUG/TRACE not audited — if Phase 1.4 ever
raises `log.level` for troubleshooting, treat logs as potentially payload-bearing
and re-run §4.1 at that level first."*

### IN-03: TLS-termination instruction is sound but lacks an explicit network-isolation precondition

**File:** `docs/fasten-admin.md:30-31, 313`

**Issue:** §0 and §6 instruct Phase 1.4 to *"terminate TLS at Traefik and keep
Fasten HTTP-internal."* This is the correct architecture and is justified (Traefik
fronts it), so it is not a security regression. But the contract states it as a
bare instruction without the load-bearing precondition: plain-HTTP Fasten is only
safe if Fasten is *not* bound to a host-reachable interface and is on an isolated
Docker network reachable only by Traefik. Without that explicit precondition a
Phase 1.4 implementer could expose plain-HTTP Fasten on a host port (the spike
itself mapped host `18081` → container `8080`, line 320-321) and carry the JWT in
cleartext.

**Fix:** Append to §0 bullet 3: *"— precondition: Fasten MUST NOT publish a host
port in prod; it must sit on an internal Docker network reachable only by Traefik.
Bearer JWTs travel that hop in cleartext, so the hop must never leave the host."*
